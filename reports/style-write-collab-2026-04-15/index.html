Что хотели

Унифицировать запись стилей инспектора между VS Code-расширением и SaaS. Сделать стратегию шеренной, framework-aware, source-aware, condition-aware и theme-aware. Два агента (Claude и Codex) должны были автономно дискутировать на таймерах и сходиться к рабочей спеке.

Что вышло

Спека получилась богатая (5 документов, ~10к строк), архитектурно цельная — есть StyleWritePlan union, framework adapter umbrella, source ownership с confidence, theme resolver, prop mappers. Codex нашёл 3 P1 регрессии в моём прошлом пасе — две из них я починил (RuntimeThemeContext проза, ScriptObjectStylePlan + mantine), одну починил частично (CssFilePlan invariants — оставил optional с runtime guard вместо discriminated union, и Codex это снова поймал в следующем ревью).

Что сломалось в коллаборации

Дискуссии не было. Был asymmetric ping-pong: Claude правит → Codex поллит и ревьюит → Claude получает явный пинг от Alex и снова правит. Codex запущен как daemon с 10-минутным polling-loop; Claude каждый раз — fresh session, инициируется только Alex'ом вручную. Никакого механизма «Codex разбудил Claude» нет. Convergence criterion отсутствует. Споров между агентами не было — каждый просто принимает фиксы другого как должное.

Главная поломка
Codex может опросить ledger в любой момент, Claude — нет. Alex был необходим как живой message bus между двумя агентами. Это не «два агента работают вместе», это «два агента работают через человека».

3. Архитектура спеки (что наработали)

Спека описывает шеренный pipeline для записи стилей: один путь для VS Code-расширения и SaaS. Три ключевых уровня — capabilities (что доступно в проекте/элементе), source ownership (кто владеет свойством сейчас), write plan (конкретная мутация).

3.1 Pipeline записи (write flow)

Manager диспатчит на один из 6 планов через развилку по sourceForm: elementClass → TailwindPlan, cssStyleRule → CssFilePlan, scriptReactStyleRule → ScriptObjectStylePlan, scriptNativeStyleRule → ScriptTemplateStylePlan, adapterKnownElementProp → AdapterPropPlan, arbitraryElementProp → ArbitraryPropPlan. Per-target value conversion (opacity 50→0.5 и т.п.) живёт в адаптерных writer'ах, не в шеренном codec'е.
Spec correction (F-010)
В первой версии спеки InspectorValueCodec отвечал за target-conversion: opacity 50 → 0.5 (CSS), 50 → '50' (Tailwind), 50 → 0.5 (Tamagui). Это неверно. Codec должен только валидировать и нормализовывать значение к inspector-form'е (например: '50%' / '50' / 50 → '50'). Конверсия в target value — забота адаптера, потому что только адаптер знает value space своего target'а (CSS-opacity 0..1, Tailwind utility scale 0..100, Tamagui prop number 0..1, ...). Один центральный target-converter — это leak адаптерных знаний в шеренный layer. См. F-010 в open items.

3.2 Source Tab routing — кто решает куда писать

Routing решает куда применить write. Два confidence-поля: routeDecision.confidence (метод) и plan.confidence (outcome) — ортогональны, не сливать.

3.3 Theme resolution — что особенного

Theme — отдельная подсистема. Differentiates:

ПонятиеЧто этоГде живёт
RuntimeThemeContextтекущее предпочтение пользователя light/dark/systemper-request на StyleWriteContext
theme conditionусловие в source: .dark &, data-theme='dark'StyleCondition.theme[]
theme value ownerгде определён цветовой токенcascadeOwner на PlainCssFilePlan
write targetчто мутировать: variable, branch, tokentarget.declarations

Ключевое правило: system — это IDE preference, не source condition. Resolver приводит system → light или dark до роутинга.

Inline fallback permanentНе временный phase guard, а универсальный fallback когда никакой adapter не может безопасно построить source-write.

5. Postmortem: почему коллаборация развалилась

Протокол был задокументирован в workprocess.md. На бумаге выглядел разумно. На практике развалился по 6 конкретным причинам.

5.1 Asymmetric agents

Поломка #1 — несимметричный контроль
Codex запущен как long-running daemon с repeat-cadence (читает ledger каждые 10 минут). Claude — fresh session, инициируется только Alex'ом. Когда Codex добавляет findings — Claude об этом не узнаёт автоматически.

5.2 No invocation channel

Поломка #2 — нет канала пинга
Codex физически не может «разбудить» Claude. Нет API, нет webhook'а, нет shared queue. Единственный способ перенести state из Codex'а в Claude — Alex вручную пишет в Claude «прочитай ledger».

5.3 Session context loss

Поломка #3 — потеря контекста между сессиями
Каждый раз когда Alex запускает Claude заново — это новая сессия, нулевой контекст. Claude должен с нуля прочитать workprocess (~1100 строк), plan.md (4400 строк), 3 sub-spec файла. Это 30k+ токенов на one-shot чтение, и часть деталей теряется.

5.4 No convergence criterion

Поломка #4 — непонятно когда коллаборация закончена
Status board: Claude `done`, Codex `done`. Но Codex добавил 2 P1 находки и пошёл спать. По формальному протоколу — work done. По факту — есть открытые блокеры. Никакого правила «не помечать done пока есть unresolved P1 findings» нет.

5.5 No discussion / disagreement protocol

Поломка #5 — нет настоящего диалога
Каждый агент принимает фиксы другого как должное. Если Claude поверхностно починил CssFilePlan (runtime guard вместо compile-time discriminated union) — Codex это заметил и снова открыл, но не написал «я не согласен с этим подходом, я бы сделал X». Просто log new finding, дальше ждать. Convergence через тишину, а не через консенсус.

5.6 Observer как message bus

Поломка #6 — Alex носит сообщения
По итогу Alex был необходимым звеном для каждого хода. «Codex что-то нашёл — расскажи Claude». «Claude починил — расскажи Codex». Это не двухагентная коллаборация. Это observer-driven sequential workflow с двумя исполнителями.

5.7 Что в этом всё-таки сработало

Что не сломалось
  • Ledger как single source of truth — workprocess.md действительно даёт обоим агентам общий контекст. Без него было бы хуже.
  • 📍 Review Session формат — позволяет легко найти где чей turn, что сделано, какие validation команды запускались.
  • Asymmetric roles (Claude — elaboration, Codex — review) — хорошая идея в принципе. Codex пойман на adversarial review, Claude — на детализации. Каждый из них находит то, что другой пропускает.
  • Версия плана v1 → v2 — несмотря на провалы протокола, спека реально стала чище: 3 регрессии нашли и (частично) починили. Без второго агента эти баги ушли бы в production-код.

6. Протокол v2 — конкретный дизайн

Цель: исключить Alex'а как message bus. Сохранить asymmetric roles. Добавить convergence criterion и disagreement handling.

6.1 Архитектура: orchestrator + структурированный ledger

Orchestrator — единственный «живой» процесс. Он читает структурированный ledger, решает чей turn, вызывает Claude или Codex headless. Observer пингуется только при disputes или convergence.

6.3 Orchestrator pseudocode

#!/usr/bin/env bash
# agent-loop.sh — runs both agents until convergence
set -euo pipefail

LEDGER=docs/specs/2026-04-14-style-write-unification-workprocess.md
FINDINGS=docs/specs/2026-04-14-style-write-unification-findings.yaml
QUIET_CYCLES=0
MAX_QUIET=3

while true; do
  open_for_claude=$(yq '[.findings[] | select(.status=="open" and .fix_owner=="claude")] | length' "$FINDINGS")
  open_for_codex=$(yq '[.findings[] | select(.status=="open" and .fix_owner=="codex")] | length' "$FINDINGS")
  disputes=$(yq '[.findings[] | select(.status=="dispute")] | length' "$FINDINGS")

  if [[ $disputes -gt 0 ]]; then
    notify_observer "disputes need adjudication: $disputes"
    break
  fi

  prev_count=$(wc -l < "$LEDGER")

  if [[ $open_for_claude -gt 0 ]]; then
    claude -p "Read $LEDGER and $FINDINGS. Address all findings where fix_owner=claude AND status=open. Mark each closed in findings.yaml. Append a 📍 Review Session turn to $LEDGER."
  elif [[ $open_for_codex -gt 0 ]]; then
    codex exec "Same instruction, fix_owner=codex."
  else
    # Both sides clear → run codex review of last claude turn
    codex exec "Review the most recent 📍 turn by claude in $LEDGER. If you find new findings, append them to $FINDINGS with raised_by=codex."
  fi

  new_count=$(wc -l < "$LEDGER")
  if [[ $new_count -eq $prev_count ]]; then
    QUIET_CYCLES=$((QUIET_CYCLES + 1))
    [[ $QUIET_CYCLES -ge $MAX_QUIET ]] && { notify_observer "convergence reached"; break; }
  else
    QUIET_CYCLES=0
  fi

  sleep 60
done

6.5 Что НЕ решается v2

Honest limitations
  • Стоимость токенов — каждое invocation Claude перечитывает ledger + plan.md = ~30k токенов на холодный старт. Для 20 циклов это 600k токенов только на чтение. Можно митигировать кэшированием prompt'а на API-уровне.
  • Stuck loops — если Claude и Codex не могут договориться (Claude чинит, Codex re-opens, бесконечно), orchestrator должен после N итераций по одному finding'у эскалировать observer'у. В pseudocode это не показано — нужно добавить counter per-finding.
  • Race conditions — если Claude и Codex вызываются параллельно (не sequential как в pseudocode выше) — могут одновременно править ledger. Sequential invocation решает, но удваивает latency.
  • «Настоящая дискуссия» — даже в v2 нет real-time back-and-forth. Это всё ещё turn-based ledger workflow. Real conversation между LLM-агентами — отдельная нерешённая проблема.
